2 · How work flows

Todo→ In Progress→ In Review→ Done
  • Todo — picked + scoped, not started.
  • In Progress — someone is building it (they assigned themselves + moved the card).
  • In Review — a PR is open; CI is running and it's waiting for Claude to integrate + merge. (This is the integrator's queue.)
  • Done — merged into main (which auto-deploys to production).
A task = a GitHub Issue. On the board it carries fields you can group, sort & filter by — Batch (which batch) · Lane (kind of work) · Arc (feature group) · assignee (who) · Milestone (stage) · Status (column). The fields + the views that use them are in §4.

4 · The board & branches — the mechanic

Cards are Issues and PRs — not raw branches. A branch reaches the board through its PR, and the PR attaches to its issue with one line in the PR description:

Closes #15  →  the PR links to issue #15 (shown in the card's "Linked pull requests" field), and merging it closes the issue.

5 · One pattern we all follow

  • Branch off latest main: pos/<id>-slug (e.g. pos/17-restock). Pull main first.
  • Commit: [DD-XX] / [Wave NN] / [fix] / [chore] one line + a body with the why and any migration / env / risk note.
  • PR: always write Closes #N. CI (typecheck · lint · test · build) must be green.
  • Merge: self-merge is OK, but ping the other dev on risky areas — auth · RLS · money · inventory · migrations.
  • Migrations (schema/SQL): add the file under database/migrations/ + a ledger note, then a human applies it in the Supabase SQL editor (Claude can't run database changes).
  • Two people coding at once → separate git worktrees (own folder per session) so files never collide.

6 · Merging & conflicts (Claude = integrator)

Parallel branches get merged by Claude, who keeps main green.

  • Textual / compatible conflict (you both edited nearby, or added different things) → Claude resolves it, keeping both features, re-runs CI, and pings the author to confirm their part survived.
  • Behaviour clash (both changed the same thing differently) → that's a product decision, not a merge → Claude brings it to the co-founders to decide (product call led by visanchan).
A merge is "done" only when all three agree: Claude's resolution + the author's quick look + green CI.  resolve → confirm → merge. No blind merges.

7 · Keep the board honest — checklist

  • ☐ Turn on the three workflows (§4) — one time.
  • ☐ Every task is a GitHub Issue with a Milestone + area + assignee.
  • ☐ Every PR says Closes #N and goes green on CI.
  • ☐ Grab work = assign yourself + drag the card to In Progress.
  • ☐ Coding in parallel = separate worktrees.
